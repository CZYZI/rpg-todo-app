# PRD：待办事项 → RPG冒险任务转化 Web 应用

**版本**：v0.1 · **日期**：2026-05-09 · **作者**：许清楚（产品经理）

---

## 1. 产品目标

> 将枯燥的日常待办事项转化为 RPG 冒险任务，通过积分与皮肤激励机制提升用户任务完成动力，让效率管理变得有趣而持续。

---

## 2. 目标用户与场景

| 维度 | 描述 |
|------|------|
| **核心用户** | 18-35岁，有轻度拖延倾向、喜爱游戏或二次元文化的效率工具使用者 |
| **次要用户** | 学生群体（作业/考试任务管理）、自由职业者（项目拆解） |
| **使用场景 1** | 早晨整理今日待办，导入后查看生成的"冒险任务"，获得开启一天的仪式感 |
| **使用场景 2** | 完成任务后标记完成，积累积分并兑换喜爱的皮肤，形成正向反馈闭环 |
| **使用场景 3** | 选择"赛博朋克"世界观，在霓虹特效界面中处理技术类任务，沉浸感强化专注度 |

---

## 3. 用户故事

| # | 角色 | 故事 | 验收标准 |
|---|------|------|---------|
| US-01 | 新用户 | 我想要粘贴文本导入待办，快速体验转化效果，而不需要注册账号 | 无账号直接使用，粘贴后可一键生成任务卡片 |
| US-02 | 普通用户 | 我想要选择世界观风格，让生成的任务描述符合我的喜好 | 导入时必选世界观（3选1），生成内容与风格一致 |
| US-03 | 普通用户 | 我想要看到任务的积分详情，了解完成任务能获得多少奖励 | 任务卡片清晰展示基础分、加成分、总积分 |
| US-04 | 活跃用户 | 我想要通过连续完成任务获得额外奖励，保持使用习惯 | 连续3天完成任务触发1.2倍连击加成，有视觉反馈 |
| US-05 | 积分用户 | 我想要用积分兑换皮肤，个性化我的界面 | 商店展示皮肤预览，兑换后CSS即时生效 |
| US-06 | 回访用户 | 我想要每天登录都有保底奖励，维持参与感 | 每日首次打开应用自动发放50积分 |
| US-07 | 所有用户 | 我想要任务生成不重复，避免每次看到一样的描述 | 同一任务再次生成时，AI替换怪物名称/场景元素 |
| US-08 | 注重隐私的用户 | 我想要确认我的待办内容不被上传存储，保护个人隐私 | 原始待办仅前端处理，API调用后不持久化原文 |

## 4. 需求池

### P0 — MVP 必须实现

| ID | 功能点 | 说明 |
|----|--------|------|
| P0-01 | 文本粘贴导入 | 支持多行文本批量粘贴，每行识别为一条任务 |
| P0-02 | 世界观选择（3种） | 中世纪奇幻 / 赛博朋克 / 现代冒险，导入时必选 |
| P0-03 | 通义千问任务生成 | 调用 qwen-max，按 Prompt 模板生成20字内RPG任务描述 |
| P0-04 | 任务卡片展示 | 显示RPG描述、积分详情（基础分+加成）、完成按钮 |
| P0-05 | 基础积分计算 | 基础积分 = 耗时（分钟）× 难度系数（默认1.0） |
| P0-06 | 任务完成标记 | 手动点击完成，积分计入localStorage |
| P0-07 | 皮肤兑换商店 | 主题皮肤3档（500/2000/5000分），兑换后CSS即时生效 |
| P0-08 | 积分持久化 | 积分与解锁记录存 localStorage，永不重置 |
| P0-09 | 防重复生成 | 相同任务二次生成时 Prompt 注入历史描述，要求替换元素 |
| P0-10 | 每日登录50保底积分 | 按日期判断，每日首次访问自动发放 |

### P1 — 迭代一优先

| ID | 功能点 | 说明 |
|----|--------|------|
| P1-01 | CSV/TXT 文件上传 | 解析文件内容，同文本导入逻辑 |
| P1-02 | 连击加成 | 连续3天完成任务触发1.2倍乘数，需记录每日完成日期 |
| P1-03 | 个人主页 | 等级进度条（每1000分升1级）、皮肤收集进度 |
| P1-04 | 世界观动态主题 | 任务列表背景/字体随世界观切换（赛博朋克配霓虹特效） |
| P1-05 | 角色皮肤（5款） | 300-1500分，基础款免费，兑换后在个人主页展示 |
| P1-06 | 功能道具：双倍积分卡 | 1000分兑换，激活后1小时内完成的任务积分×2 |
| P1-07 | 自定义任务耗时 | 任务卡片支持用户手动输入耗时（分钟），影响积分计算 |

### P2 — 后续版本

| ID | 功能点 | 说明 |
|----|--------|------|
| P2-01 | 成就系统 | 首次完成特定任务+500分，成就徽章收集 |
| P2-02 | Todoist 等 API 集成 | 第三方任务平台OAuth接入 |
| P2-03 | 难度系数自动调整 | 基于历史完成数据动态更新系数 |
| P2-04 | 动态音效 | 不同世界观背景音乐与完成音效 |
| P2-05 | 防作弊升级 | 检测异常快速完成行为（如<10秒完成标记的任务）并降权处理 |

## 6. 潜在风险清单

| # | 风险 | 影响 | 应对策略 |
|---|------|------|---------|
| R-01 | **过度游戏化导致忽视真实任务** | 用户沉迷积分/皮肤，反而不执行实际任务 | 任务完成需手动标记（非自动），强调"完成真实任务才获积分"；界面文案始终呈现原始任务名称，防止用户"只看RPG描述忘了做什么" |
| R-02 | **API Key 暴露风险** | 纯前端调用通义千问API，Key 在客户端可见，存在盗用风险 | 初期MVP阶段用户自行输入Key（本地存储），或通过BFF层代理；上线前明确Key保护策略，提示用户设置用量上限 |
| R-03 | **AI生成内容质量不稳定** | 生成的任务描述语义不通、与世界观不符或内容敏感 | 前端做基础输出过滤（长度、关键词检测）；Prompt中添加"禁止生成不当内容"约束；预设10条兜底描述模板以防API失败 |
| R-04 | **localStorage 数据丢失** | 用户清除浏览器数据导致积分、皮肤全部清空，严重影响留存 | 页面显示"积分仅存本地，清除浏览器数据会丢失"的醒目提示；P1阶段预留账号体系接口，支持数据云同步 |
| R-05 | **积分系统被刷分滥用** | 用户批量导入大量短任务并快速点击完成，无限刷积分 | 设置单任务最低耗时下限（如5分钟）；检测1分钟内连续完成超过5个任务时触发冷却提示；单日积分上限（如1000分） |

## 7. 待确认问题

| # | 问题 | 决策方向 | 优先级 |
|---|------|---------|--------|
| Q-01 | **API Key 托管方案**：MVP 是否要求用户自带 Key，还是由产品方统一代理（需后端）？ | 建议 MVP 用户自带 Key，降低初期成本，但需评估用户接受度 | 高 |
| Q-02 | **任务耗时默认值**：单条任务默认30分钟是否合理？是否允许导入时批量设置耗时？ | 需用户调研，建议提供"快速/标准/深度"三档预设（15/30/60分钟） | 中 |
| Q-03 | **数据持久化边界**：隐私保护要求"不存储原始数据"，但防重复机制需要存储历史生成记录，如何界定？ | 建议仅存储生成后的RPG描述（脱敏），不存储原始待办文本 | 高 |
| Q-04 | **积分单日上限**：是否设置每日积分上限以防刷分？如设置，阈值如何确定？ | 建议设 800-1000分/日上限，具体值需结合正常使用行为估算 | 中 |
| Q-05 | **世界观扩展计划**：初期3种是否足够？后续是否以付费/积分解锁新世界观作为变现路径？ | 需产品方明确商业化方向，建议 P2 阶段以积分解锁作为留存手段 | 低 |
